cookie&session

cookie是浏览器端的本地存储空间，和localstorage和sessionstorage一样都是浏览器端本地存储的一部分

cookie主要提供服务器端存储的数据

作用：用来存储一些数据

cookie存储的特点：

  • 1:cookie是按照域名来存储的，这点很傻，不智能，在哪一个域名下存储的cookie就在只能在当前域名下访问，换一个域名就不能访问

    比如本地：localhost和127.0.0.1 虽然是同一个网站但是是俩名，cookie存的时候是按名存的，他不管你打开的是不是一个网站，只要是俩名，他就访问不了

  • cookie按照文件路径存储

    你在a文件夹存储的数据，只能在a文件夹及子文件夹访问，在a文件夹的上一级目录则不能访问

    （可以直接在控制台里的cookie列表里的path设置存储数据）

  • cookie的存储是按照字符串的形式存储

    真实的存储形态是：’key=value; key2=value2; key3=value3’

    在控制台显示的是浏览器帮我们格式化后的展现形式为了看着方便

  • 存储大小

    4kb左右

    50条左右 经测试，目前的谷歌浏览器在150条左右，但是官方发布的规则还是50条左右

  • 实效性 ：cookie中的数据是有过期时间的，超过过期时间数据会被浏览器自动删除

    默认是会话级别的时效（关闭浏览器就没有了）

    我们可以手动设置cookie的时效（关闭浏览器以后依旧保存，比如7天免登录，就是将一些信息存储在cookie里了，不管是否关闭浏览器，在7天内都可以自动登录，但是存的信息不是用户名和密码，我们在浏览器的设置里可以清除cookie，清除以后我们一些免登录信息就没有了）

    如果不给cookie设置过期时间，那么cookie在关闭的时候cookie就会被删除

  • 操作权限

    前端可以操作，后端也可以操作，cookie是前端本地的一些东西，但是前端可以用后端也可以用

    localstorage和sessionstorage只能前端操作，后端操作不了

  • 请求自动携带 ：cookie中的数据会随着请求自动发送到服务器端

    只要cookie空间里面有数据，那么你在发送任何一个请求的时候，自动携带，就是我们不需要操作，浏览器在组装请求报文的时候会自动把cookie里的东西给带到服务器

    以前有一个攻击叫xss跨站脚本攻击，是攻击别人网站的，就是使用cookie注入即在cookie里写入一些东西然后只要向服务器发送请求，就带到服务端了

前端操作cookie

前提：想要使用或者操作cookie，页面必须在服务器上打开，因本地打开没有域名，cookie是按照域名存储的，没有域名就存不上cookie

  • 前端操作cookie的api只有：document.cookie

document.cookie是一个读写的属性

​ - 读：document.cookie

​ 会把cookie空间里面所有数据以字符串的形式一次性全部拿出来

​ - 写：document.cookie =’key=value’

​ 写入一条cookie ，一次只能写入一条cookie

cookie是不能删的，到达过期时间自动就没有了

设置cookie（重点）

  • 一次只能设置一条cookie

  • 在设置的时候，每一条的分号后面可以书写对本条cookie的描述

    • expires 过期时间

      如果不进行特殊设置，默认是会话级别的时效性 session

    如果想要修改：就需要在写入cookie的时候，在键值对后面写上expires描述。

    expires=时间对象

    或者

    expires=时间对象的字符串

    案例：设置一个有时效性的cookie

    分析：我们获取的时间对象是终端的时间（客户端），不是服务器的时间，服务器的时间是世界标准时间

    中国标准时间北京时间比世界标准时间快（早）8个小时

    但是我们设置cookie的时候，不管我们给他什么时间节点，他都当做世界标准时间使用

    需求：我们要设置一个30s以后过期的cookie

    步骤：获取当前时间对象，先向前调整8个小时，准换为世界标准时间，再向后调整30s。 然后将调整后的时间设置给cookie，这样cookie会将他当做世界标准时间来使用，这条cookie真正的过期时间是中国标准时间的30s后过期

    • path 存储路径

      如果你不进行path的特殊设置，会默认按照当前的目录结构进行存储，也就是说我在当前文件设置cookie只能在当前的文件夹下访问，在上一级文件夹不能访问

      如果想要修改 ，就需要在写入cookie的时候，在键值对后面 设置上你要存储的地址

      document.cookie=’key=value;path=你要存储的路径’

    • 在设置的时候不管存储的是什么数据类型，都会自动转换成字符串的格式存储

      所以cookie一般都是存储一些简单数据，帮我们做一些简单记录信息用的

    • 在设置的时候，如果设置同名的cookie就是修改

      注意：一定是要是相同的路径下同名个cookie才是修改

    • 在设置的时候，如果设置一个cookie的过期时间在当前时间的以前，就是删除这条cookie

      因为在到达过期时间的那一瞬间，这一条cookie已经从浏览器移除了

      所以删除cookie就是这样操作的

cookie只是帮我存储一些简单的信息，不做其他很复杂的操作

node中怎么使用session

cookie的问题

cookie 虽然很方便，但是使用 cookie 有一个很大的弊端，cookie 中的所有数据在客户端就可以被修改，数据非常容易被伪造，那么一些重要的数据就不能存放在 cookie 中了，而且如果 cookie 中数据字段太多会影响传输效率。为了解决这些问题，就产生了 session，session 中的数据是保留在服务器端的。

session原理

sessionStorage，localstrory，cookie都是客户端的存储空间，

session:

session实际上就是一个对象，存储在服务器端的内存中，在session对象中也可以存储多条数据，每一条数据都有一个sessionid作为唯一的标识

cookie和session都是用来存储数据的，cookie在客户端，session在服务器端

一个存在于服务器端的存储空间，当打开session空间的的那一个瞬间，就会生成一个密码，这个密码会自动存储到cookie里，等到返回前端的时候，会自动将cookie带回去

同源策略

  • ajax请求的限制

目前我们学习的是客户端发送请求，实际是同一个网站的客户端向同一个网站的服务器端发送请求。

但是工作中我们会遇到，自己需要的数据在别人家的服务器上，我们需要别人家服务器端提供的请求地址获取我们需要的数据，比如获取天气信息，我们就需要通过国家气象局提供的请求地址来获取，但是ajax只能向自己的服务器端发送请求，请求别家的服务器是不被允许的

Ajax只能向自己的服务器发送请求，比如现在有一个A网站，一个b网站，a网站的html文件只能向a网站服务器中发送ajax请求，b网站的html文件只能向b网站的中发送ajax请求

这就是ajax请求的限制，这是因为浏览器有同源政策的限制，这是浏览器给ajax的限制。它不允许ajax向自己以外的其他服务器发送请求

  • 同源

如果两个页面拥有相同的协议，域名和端口，那么这两个页面就属于同一个源，即这两个页面来自同一个服务器。三个相同中，只要一个不相同，就不是同源

跨域：只要协议，域名，端口有一个和服务器不相等，ajax请求就会报CORS错误 cors（跨域资源共享错误）

解决跨域问题

  • 同源政策存在的目的

同源政策是为了保证用户信息的安全，防止恶意的网站窃取数据，最初的同源政策是指a网站在客户端设置的cookie在b网站不能访问（如果没有同源策略：cookie 是用来保存用户的登录状态的，如果用户刚登录过某银行的网站且没有退出登录就去浏览其他的网站，其他网站就可以拿到银行网站的用户的cookie，冒充用户为所欲为）

随着互联网额发展，同源政策也越来越严格，在不同源的情况下，其中有一项就是无法向非同源地址发送ajax请求，如果请求，浏览器就会报错，其实请求是可以发送过去的，只不过浏览器拒绝接收服务器返回的结果，最终请求还是失败

解决同源限制的问题

（绕过浏览器的同源政策）

- 使用jsonp解决同源限制问题

jsonp是json with padding 的缩写，他不属于ajax请求，但他可以模拟ajax请求

这需要前后端人员配合完成 ，步骤：

  • 1:将不同的服务器请求写在script标签的src属性

    script 标签的src属性可以发送请求且不受同源政策的影响，所有可以写非同源的请求地址

注意：

script标签主要是用来加载外部js文件的，请求地址以.js结尾；但是其实script标签的请求地址可以是任意形式的，不是必须以js结尾，但是请求地址不论什么样子，但是它必须返回合法的js代码，因为script标签在加载了请求地址中的内容以后，要将内容作为js代码来执行

  • 2:服务器响应数据必须是一个函数的调用，真正要发送给客户端的数据需要做为函数调用的参数

    当浏览器通过script标签将响应内容加载完成以后，会自动将响应内容作为js代码，即响应内容加载完以后会立即调用这个函数，因为加载过来的内容就是函数调用的代码

    在服务器端响应内容必须是字符串，在字符串中包裹函数调用的代码，因为如果不是字符串就是真正的函数调用，那么这个函数就会在服务器端执行，但我们的目的是让函数在客户端执行

    在服务器端返回函数调用的代码的同时还需要将客户端真正需要的数据写在函数掉用实参的地址，因为函数在客户端被执行的时候，客户端就可以通过形参来接收这个实参数据了，也就是说，客户端可以通过函数的形参拿到服务器端返回的真正的数据了，但是有一点需要注意，函数是在客户端被调用的，所以在调用函数之前要先准备好这个函数的定义，即有函数才可以调用起来

  • 3:在客户端准备好函数的定义

    注意：定义这个函数要在全局作用域写，且在script标签的上面，这样在调用函数的时候才可以找到这个函数

注意：其实就是将函数定义在客户端，在服务器端返回函数调用的代码，虽然服务器返回的是一个字符串类型的函数调用，但是由于这个代码被写在了script标签中，所以这个字符串类型的代码在客户端会被当做真实的js代码执行

jsop本质含义就是，将json数据作为填充内容，即在服务器端将json数据作为函数的参数，将json数据填充到函数中

- 访问非同源数据 服务器端解决方案

同源政策是浏览器给予ajax技术的限制，服务器端是不存在同源政策限制的，服务器端开发语言可以直接访问非同源网站的数据，所以客户度当想要访问非同源网站的数据的时候可以先让自己的网站的服务器端去获取非同源网站中的数据，然后再将获取到的数据响应到自己网站的客户端，这样就绕过了浏览器的同源政策限制

如何在服务端请求其他服务器的数据，需要使用以第三方模块名字为： request （查看npm request 文档有使用方法）
